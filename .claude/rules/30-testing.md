# Testing Rule

このプロジェクト（ブラウザゲーム）のテスト方針を定義します。
ゲームは「純粋ロジック」と「副作用（描画・入力・音・DOM）」に分かれます。
**純粋ロジックは自動テスト、副作用はブラウザで手触り確認** を基本にします。

## スタック

| レイヤー | フレームワーク | ランナー |
|----------|---------------|---------|
| 純粋ロジック(JS) | Node 標準の `node:test` + `node:assert`（依存追加なし） | `node --test` |
| 副作用・統合 | ブラウザ目視 / `webapp-testing`(Playwright) | 手動・スクリーンショット |

---

## テスト優先順位

```
1. combo      （コンボ計数・減衰・派生倍率）← 最重要・純粋関数
2. collision  （円ベースの当たり判定・ボム半径ヒット）
3. bomb       （溜め時間→半径、コンボ→溜め速度 の計算）
4. stage      （難易度カーブ・ウェーブ進行の決定的な部分）
```

---

## 1. combo（コンボ）← 最重要

純粋関数なのでモック不要。実ロジックをそのままテストする。

### テスト観点
- 撃破でコンボが増える / 同時撃破（ボム）で大きく増える
- 一定時間撃破が無いとコンボが減衰・リセットされる（境界のタイミング）
- コンボ段階 → ボム溜め短縮率 / スコア倍率 / 音楽レイヤー段階 の写像が単調で範囲内
- コンボ 0 / 異常入力でも例外を投げず妥当な既定値を返す

---

## 2. collision（当たり判定）

### テスト観点
- 2 円の重なり判定（接触ちょうど・わずかに離れる の境界）
- ボム衝撃波（中心＋半径）と敵の当たり（リング内/外/縁）
- 弾と敵、敵と自機の当たり

---

## 3. bomb（チャージ計算）

### テスト観点
- 溜め時間 t → 半径 r の単調増加・上限クランプ
- コンボ C → 溜め速度の上昇（C=0 で基準値、増加で短縮）
- 溜め 0 で離した場合の最小挙動（暴発・NaN を出さない）

---

## ファイル配置

```
tests/
  combo.test.mjs
  collision.test.mjs
  bomb.test.mjs
src/
  combo.js        ← DOM/Canvas 非依存の純粋関数を export
  collision.js
  bomb.js         ← 計算部分を純粋関数として分離
```

純粋ロジックは `export` して `import` でテストする。描画・入力・音・`requestAnimationFrame` を
純粋ロジックモジュールに混ぜない（テスト不能になるため）。

---

## 副作用・統合の確認（自動テストの代わり）

`node:test` では DOM/Canvas/WebAudio を扱わない。以下はブラウザで確認する。
- `python3 -m http.server` で配信し、左連射・コンボ・ボム溜め→範囲一掃・難易度上昇・GAME OVER を目視。
- 必要なら `webapp-testing`(Playwright) で起動・スクリーンショットし、ネオン描画と HUD を確認。

---

## カバレッジ目標

| スコープ | 目標 |
|---------|------|
| combo | 90% 以上 |
| collision / bomb | 80% 以上 |

カバレッジは目安。数値を上げるためだけのテストは書かない。

---

## エージェントへのルール

- **純粋ロジック（combo/collision/bomb 計算）はモック禁止**。そのまま呼び出す。
- 純粋ロジックは副作用（DOM/Canvas/WebAudio/rAF）から分離して書く。
- 境界値・異常値は `node:test` の複数ケースでまとめる。
- テスト名は `<対象>_<状態>_<期待結果>` 形式にする。
  - 例: `combo_decay_window_elapsed_resets_to_zero`
- CI（GitHub Actions）で `node --test` が通ることを必須とする。ローカルのみ通過は不可。
