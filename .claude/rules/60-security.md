# Security Rule

ブラウザ・ゲーム（静的配信・サーバなし・秘密情報なし）向けの、最小限のセキュリティ指針です。
スクレイピング・PII・サーバ秘密情報は本プロジェクトに存在しないため扱いません。

## 既存ルールとの分担

| ファイル | 担当領域 |
|---------|---------|
| `40-deploy-ci.md` | 公開（GitHub Pages）・CI |
| `50-agent-safety.md` | エージェント操作の制約・直書き禁止 |
| **本ファイル** | **依存・秘密情報・入力（localStorage / config 値）** |

---

## 1. 秘密情報を持たない

- このゲームは API キー・トークン・サーバ秘密情報を**一切持たない**設計を保つ。
- コード・リポジトリに秘密情報を書かない。必要が生じたら、まず設計を見直す（静的配信では秘密を隠せない）。

## 2. 依存ゼロを保つ

- 外部ライブラリ・バンドラ・フレームワークを安易に追加しない（攻撃面・保守コストの増加）。
- どうしても必要な場合のみ、用途・代替不可の理由を提示してから追加する。
- 追加した場合はバージョンを固定し、`npm audit` 等で既知脆弱性を確認する。

## 3. 入力・保存データの扱い

- `localStorage`（ハイスコア等）から読んだ値は信頼しない。`Number()`/範囲チェックで検証し、壊れていれば既定値にフォールバックする（例外で停止させない）。
- ユーザー名等の文字列を画面に出す場合は `textContent` を使い、`innerHTML` に外部文字列を渡さない（XSS 回避）。
- 外部から得た文字列を `eval` / `Function` に渡さない。

## 4. 公開時の posture

- `README.md` に操作・遊び方・免責、`LICENSE`（MIT）を同梱する。
- スコア等の保存はクライアント（localStorage）のみ。サーバへ送信しない。

---

## エージェントへのルール

- 秘密情報をコード・リポジトリに置かない（持たない設計を維持）。
- 依存を増やすときは理由を提示し、脆弱性を確認してからコミットする。
- `localStorage` / 外部由来の値は検証してから使う。`innerHTML`・`eval` に外部文字列を渡さない。

---

## セキュリティチェックリスト（実装前の自問）

```
[ ] 秘密情報をコードに書いていないか？
[ ] 依存を不要に増やしていないか？増やすなら脆弱性を確認したか？
[ ] localStorage / 外部値を検証してから使っているか？
[ ] 外部文字列を innerHTML / eval に渡していないか？
```
